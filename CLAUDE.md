# Team Work Test

This project demonstrates parallel feature development using a multi-agent team architecture.

## Purpose

Show how multiple independent feature teams can work simultaneously on separate git branches,
each team progressing through a fixed pipeline: Plan → Develop → Review → PR.

## Agent Roles

| Agent | File | Responsibility |
|-------|------|----------------|
| Team Leader | `team-leader` | Receives feature requests, spawns Feature Teams in isolated worktrees, tracks all in-flight work |
| Feature Team | `feature-team` | Runs the full pipeline for one feature: coordinates Planner → Developer → Reviewer, then opens a PR |
| Planner | `planner` | Breaks a feature description into a concrete, file-level implementation plan (`PLAN.md`) |
| Developer | `developer` | Implements every task in `PLAN.md`, commits after each logical unit of work |
| Reviewer | `reviewer` | Reviews all changes, requests fixes from Developer if needed, approves when ready |

## How to use

Tell the Team Leader what feature to implement and which feature file to read:

> "Implement feature 001 — read features/001-project-scaffolding.md"

The Team Leader spawns an isolated Feature Team in the background. You can immediately request
another feature — teams work in parallel on separate branches.

## Feature tracking

All features are listed in `features/FEATURE_LIST.md`.
Each Feature Team is responsible for checking off its feature in that file as part of its PR.
The checklist update must be committed on the feature branch and included in the same PR as the implementation.

## Branch convention

`feature/<short-kebab-description>` — max 5 words, derived from the feature title.

## PR convention

- Title: `feat: <feature title>`
- Body: summary of changes + test plan (generated by the Reviewer)
- Every PR must include a commit that marks the feature as done in `features/FEATURE_LIST.md`

## Workflow

1. User tells Team Leader which feature to implement.
2. Team Leader reads the corresponding file in `features/`, spawns a Feature Team in a background worktree on a new branch.
3. User can immediately request another feature — a second team starts in parallel.
4. Each team is fully isolated; branches never conflict.
5. When a Feature Team finishes, it checks off the feature in `FEATURE_LIST.md`, opens a PR, and the Team Leader reports the link.

## Dependency order

Features 002–020 all build on 001 (Project Scaffolding). Always implement 001 first.
After 001 is merged to main, independent features can be developed in parallel.
